# HTTP Connection Contamination

# 0x01 核心机制

HTTP 连接污染的核心在于两个标准特性的“非预期碰撞”：

1. **浏览器端的连接合并 (Connection Coalescing)**：
   
   根据 RFC 7540 (HTTP/2) 和 RFC 9114 (HTTP/3)，如果两个域名解析到相同的 IP 地址，且服务器提供的 TLS 证书同时涵盖这两个域名（常见于**通配符证书 `\*.example.com`** 或 **多域名证书 SAN**），浏览器为了减少握手开销，会将原本属于不同域名的请求复用在同一个 TCP/UDP 连接上。

2. **服务端的首次请求路由 (First-Request Routing)**：
   
   为了极致的性能，某些高性能负载均衡器或反向代理在处理连接时，仅解析该连接上的**第一个请求**（如请求 A 的 `Host: blog.example.com`）来决定转发给哪个后端集群。一旦路由确定，该连接后续的所有请求（Stream）都会顺着“已建立的管道”发往同一个后端，不再重新解析 `Host` 头。

# 0x02 现实场景：组织内部的“低安全区”污染“高安全区”

在企业内网或大型云架构中，这种情况非常普遍：

### 1. 基础环境

- **通配符证书**：企业为了管理方便，给所有子域名部署了 `*.example.com`。
- **共享基础设施**：`wordpress.example.com`（低安全需求，可能存在已知漏洞或插件漏洞）和 `secure.example.com`（高安全需求，存储敏感数据）都挂载在同一个反向代理后面，且解析到同一个 IP。

### 2. 攻击路径

1. **诱导访问**：攻击者诱导已登录 `secure.example.com` 的管理员访问 `wordpress.example.com`（例如通过一个简单的链接或图片标签）。
2. **建立受污连接**：浏览器与服务器 IP 建立连接。第一个请求发往 `wordpress`，反向代理将此连接路由到 **WordPress 后端集群**。
3. **触发合并**：随后，攻击者利用脚本（如 `fetch`）在后台发起对 `secure.example.com` 的请求。浏览器发现证书和 IP 匹配，直接复用刚才发往 WordPress 的连接。
4. **跨站污染**：反向代理根据“首次请求路由”逻辑，将原本发往 `secure`（高安全区）的敏感请求，错误地转发到了 `wordpress`（低安全区）的后端。
5. **漏洞利用**：如果 WordPress 存在一个简单的反射型 XSS，攻击者就可以在 `wordpress` 的响应中注入脚本。由于浏览器认为这个响应是 `secure.example.com` 返回的，该脚本便获得了 `secure` 域下的执行权限，从而窃取其 Cookie 或执行管理操作。

# 0x03 HTTP/3 (QUIC) 的放大效应

HTTP/3 引入了更激进的连接复用策略，进一步提升了该攻击的成功率：

- **放宽 IP 限制**：在某些实现中，即便 DNS 解析出的 IP 不同，只要通过 `Alt-Svc` 头部声明了服务迁移，且证书匹配，浏览器仍可能尝试合并连接。
- **无感知迁移**：由于 QUIC 基于 UDP 且支持连接迁移，这种污染状态可能在用户切换网络环境后依然保持。

# 0x04 检测与验证 (PoC)

要验证目标系统是否存在此问题，可以通过控制请求顺序来观察：

JavaScript

```
// 1. 触发连接建立并锁定路由到子域名 A
fetch("https://sub-a.example.com/trigger", { mode: "no-cors" }).then(() => {
    // 2. 在同一连接上请求子域名 B
    // 如果 sub-b 的请求收到了 sub-a 后端的报错或特定响应，则存在污染
    fetch("https://sub-b.example.com/sensitive-data");
});
```

在 **Burp Suite** 或 **Wireshark** 中，重点观察不同虚拟主机的请求是否共享了同一个 `Connection ID` 或 TCP 流。

# 0x05 深度防御建议

| **层面**        | **措施**           | **描述**                                                                          |
| ------------- | ---------------- | ------------------------------------------------------------------------------- |
| **反向代理 (核心)** | **禁用流级路由缓存**     | 配置 Nginx (`proxy_set_header Host $host`) 或类似中间件，确保对 HTTP/2 的每个 Stream 重新进行路由决策。 |
| **证书管理**      | **避免大范围通配符**     | 对高安全等级的业务使用独立证书，不与通用业务共享 `*.example.com`。                                       |
| **网络隔离**      | **IP 分离**        | 将敏感业务解析到独立的 VIP（虚拟 IP），从物理/逻辑层断开浏览器合并连接的前提条件。                                   |
| **协议配置**      | **禁用 HTTP/2 合并** | 在反向代理侧通过不发送相同的 IP 集合或使用不同的端口来规避。                                                |
